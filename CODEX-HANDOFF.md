## Architecture

### How the site works

1. Blog posts are written as **markdown files with YAML frontmatter** in `content/`
2. `scripts/build.mjs` reads all markdown, converts to HTML, and outputs to `site/`
3. The `site/` directory is committed to the repo (it IS the deployable output)
4. On push to `main`, GitHub Actions runs the build and deploys via GitHub Pages
5. The build injects Supabase credentials from GitHub Secrets at build time
6. Repo-aware URLs are derived from workflow env (`GITHUB_OWNER`, `GITHUB_REPO`) or fall back to `kols-korner`
7. Custom domains can be supplied via `CUSTOM_DOMAIN` or a committed `CNAME` file

### Build script structure (`build.mjs`)

The build script:
- Parses markdown with a custom parser (no external dependencies beyond Node.js built-ins)
- Uses `getHeaderHTML(basePath)` and `getFooterHTML()` for reusable page chrome
- Generates: homepage, posts list, individual posts, about page, tags page, subscribe page, news page, images page, videos page, music page
- Handles media: images, videos (with ffmpeg thumbnail generation), audio (with waveform generation)
- Posts with `publish: false` in frontmatter are excluded
- Frontmatter fields: `title`, `date`, `tags`, `summary`, `image`, `publish`
- Supabase URL and key are injected via environment variables: `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`
- Canonical URL generation is environment-aware, so a repo rename or custom domain does not require another hard-coded path sweep

## Design System

### Colours
- **Primary:** Red — `#dc2626` (light mode), `#f87171` (dark mode)
- **Background (dark):** Dark charcoal `#1f2121` / `#262828`
- **Background (light):** Cream `#fcfcf9` / `#fffffe`
- **Text (dark):** Light grey/white
- **Text (light):** Dark charcoal/brown
- Full colour token system in `:root` CSS custom properties

### Typography
- **Font:** Fira Sans (Google Fonts) — weights 300, 400, 500, 600, 700
- **Base font size:** 17px
- **Body line height:** 1.7
- **Headings:** Tighter line height, heavier weight

### Theme
- Dark mode by default (`data-theme="dark"` on `<html>`)
- Toggle button in header (sun/moon icons)
- Theme preference saved to localStorage

### Conventions
- **UK English** throughout (colour, favourite, organised, etc.)
- Security headers on all pages (CSP, X-Frame-Options, etc.)
- Responsive design with mobile-first approach
- Fade-in-up entrance animations (respects `prefers-reduced-motion`)

## Outstanding Issues & Next Steps

### 1. Remove RSS section from subscribe page (Quick fix)
The subscribe page (`site/subscribe/index.html`, lines 80-90) has an "RSS Alternative" section. **The user does not want RSS.** Remove the entire `<div class="subscribe-rss">` block and its CSS (`.subscribe-rss` in `styles.css`).

### 2. Newsletter (Major feature — not yet built)
The subscribe page has a form that posts to Supabase, but there's no actual newsletter sending mechanism yet.

**What the user wants:**
- **Daily digest newsletter** sent to subscribers
- **Customisable categories** — subscribers toggle which news categories they receive
- **Beautiful HTML email template** — not plain text
- **Newsletter archive on the site** — a `/newsletter/` section separate from posts
- **Showcase section** — for AI-generated images, video, and music

**Recommended approach:**
- **Supabase** for subscriber storage + preferences (already partially set up)
- **Resend** (resend.com) for email sending — 100 free emails/day, great HTML support
- **Supabase Edge Functions** to orchestrate: query subscribers by preferences, render email template, send via Resend
- The newsletter content comes from the daily digests that are already being generated

**What needs building:**
- Supabase database tables: `subscribers` (exists), `subscriber_preferences`, `newsletter_issues`
- Supabase Edge Function for sending
- HTML email template (responsive, matches site design)
- Newsletter archive page on the site
- Category preference UI on the subscribe page
- Integration with the daily digest pipeline

### 3. News gathering system (Shared tool — now an active local priority)
The current `scripts/fetch-news.mjs` uses RSS feeds with only 3 sources. This is still only a stopgap. The user wants a **shared news gatherer** covering **60-70 curated website sources**, backed by **Supabase**, and reusable across multiple websites with site-specific filtering or tags.

Current state:
- The repo-local stopgap pipeline has been repaired enough to keep publishing digests
- `.github/workflows/daily-digest.yml` now runs twice daily
- Legacy raw digests were backfilled into publishable posts during the last Codex session
- The real source list is still missing from this repo

Important direction:
- This work should now continue in a **local Codex session**, not be treated as out of scope
- The final gatherer will likely live as a shared local tool or separate repo/service, with this repo consuming its outputs
- Because the machine is always on, the real gatherer should probably run via a **local Windows Scheduled Task** rather than relying only on GitHub Actions

See `NEWS-GATHERER-HANDOFF.md` for the dedicated brief.
